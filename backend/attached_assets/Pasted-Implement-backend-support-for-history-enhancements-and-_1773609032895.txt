Implement backend support for history enhancements and cross-model conversation continuity in CaelynAI.

You only have access to the BACKEND repo.

Context:
- This backend owns all APIs, persistence, DB logic, model routing, and agent orchestration.
- The frontend is only a UI client.
- All conversation history, model continuity, ticker snapshot data, and retrieval APIs should be handled here.

## Main backend goals

### 1) Conversation continuity across model switching
A conversation must persist across:
- solo model modes
- Caelyn mode / multi-model mode
- switching models mid-conversation

Required backend behavior:
- conversation_id is the canonical thread identity
- model selection is separate from conversation identity
- switching models during an active chat must NOT create a new conversation
- any model/agent should be able to continue the same thread if the same conversation_id is active
- when generating a response, the backend must load the prior thread context by conversation_id regardless of which model previously handled the thread
- a conversation started by one model must be immediately continuable by another model without losing context

### 2) Persist ticker mention snapshots for full history/backtesting context
For assistant responses, detect ticker mentions and persist:
- ticker symbol
- timestamp of mention / response
- price at time of mention
- optional asset type/source if already available

Important:
- mention-time price must be captured and stored when the response is generated
- mention-time price must not drift later
- this stored historical price will be compared against current live price when viewing history
- do not rely on reconstructing mention-time price later

### 3) Support two history API experiences

#### A. Full History page
Return full conversation history with ticker snapshot data:
- conversation metadata
- messages
- timestamp
- model used
- persisted ticker mention data per relevant message

#### B. Right-side terminal recent history
Return only the 5 most recent conversations for quick reference:
- conversation id
- title or preview snippet
- updated_at / last activity
- optional model label if useful

Important:
- the right-side recent history API must NOT include backtesting/ticker comparison datapoints
- keep it lightweight

### 4) Support current-price comparison for history page
Because backend owns data access/APIs, provide backend support for current price lookup for persisted ticker mentions.

Preferred implementation:
- add a lightweight batch endpoint that accepts tickers and returns current prices
- deduplicate repeated tickers
- fail gracefully if one ticker cannot be resolved
- keep old history usable even if current price lookup fails
