## Main backend goals

### 1) Sector preset buttons must map to true sector-specific analysis
When a sector preset button is clicked, the backend must:
- detect the selected sector preset key reliably
- use a sector-specific query path / prompt template / response builder
- constrain the response to that sector
- avoid falling back to generic market-theme or broad watchlist behavior unless explicitly intended

Examples:
- Energy → energy sector intelligence
- Materials → materials sector intelligence
- Aerospace/Defense → aerospace/defense sector intelligence
- Tech → broad tech sector intelligence
- AI/Compute → AI/compute/infrastructure sector intelligence
- Quantum → quantum sector intelligence
- Fintech → fintech sector intelligence
- Biotech → biotech sector intelligence
- Real Estate → real estate / REIT / housing-related sector intelligence

### 2) Sector output should include the relevant sector ETF / benchmark ticker(s)
When a sector preset is clicked, the response should prominently reference the relevant sector ETF or benchmark(s), not generic macro placeholders only.

Examples:
- Energy → XLE
- Materials → XLB
- Aerospace/Defense → ITA or XAR
- Tech → XLK
- Real Estate → XLRE
- Fintech / AI / Quantum / Biotech may use best-fit ETF(s), index proxies, or curated thematic benchmark tickers where standard SPDR sector ETFs do not exist

Important:
- The sector response should not default to generic rows like SPY/QQQ/IWM/VIX with N/A everywhere as the primary focus.
- Macro context can still exist, but the lead benchmark rows should be sector-relevant.

### 3) Keep the overall rich response style, but improve the stock-selection section
The current response style is directionally good.
Keep the broader structure, but make these changes:

#### Rename:
- "Top Moves Today" → "Top Stocks to Watch"

#### Change the stock selection logic:
Show:
- 1–3 large caps
- 1–3 mid caps
- 1–3 small/micro caps

These names should be selected from the chosen sector only and should represent stocks that currently show some combination of:
- relative strength
- current or upcoming catalysts
- hype / social attention
- unusual institutional flows
- insider buying if available
- news momentum
- strong fundamentals where relevant
- strong technical setups where relevant

This should not be random. It should be an intentionally ranked “watch list” for that sector.

### 4) Preserve rich sector intelligence
When a sector preset is clicked, the response should still include rich intelligence like:
- sector verdict / bias
- why the sector matters now
- what’s moving in the sector
- top signals from different scanners/sources if available
- upcoming catalysts
- portfolio / tactical bias

But the full output must stay tied to that sector.

### 5) Improve preset routing / response typing
Audit the current preset-button backend logic and identify:
- how sector preset buttons are mapped
- why they are producing generic theme output
- where preset keys are converted into internal scan types or prompt payloads
- where labels like `ai_compute` are coming from

Fix this so that:
- sector preset buttons use the correct sector-specific route/template/formatter
- response metadata is sector-friendly
- the displayed title/identifier is human-friendly, not raw internal slug text unless intentionally desired
