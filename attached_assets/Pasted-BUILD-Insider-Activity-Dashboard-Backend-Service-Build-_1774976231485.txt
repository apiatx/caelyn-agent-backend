BUILD: Insider Activity Dashboard — Backend Service

Build a complete backend service that fetches SEC Form 4 insider trading filings, scores each transaction by how unusual/significant it is, stores results in our Neon PostgreSQL database (30-day retention), and exposes REST API endpoints for the frontend.

=== STEP 1: Install the edgartools Python library ===

Run this in the shell:
pip install edgartools --break-system-packages

edgartools is a FREE, open-source Python library that talks directly to SEC EDGAR. No API key needed. No rate limits. It parses Form 4 filings into structured Python objects.

You MUST set a User-Agent identity for SEC compliance. Add this at the top of the service file:
from edgar import set_identity
set_identity("CaelynAI Trading Dashboard admin@caelynai.com")

=== STEP 2: Create the database table ===

Create a new file: services/insider_activity_service.py

Add a PostgreSQL table called "insider_transactions" with these columns:
- id: SERIAL PRIMARY KEY
- accession_number: VARCHAR(30) UNIQUE (the SEC filing ID — use this to avoid duplicates)
- ticker: VARCHAR(10)
- company_name: VARCHAR(255)
- insider_name: VARCHAR(255)
- insider_title: VARCHAR(100) (CEO, CFO, Director, 10% Owner, etc.)
- transaction_type: VARCHAR(20) (Buy, Sale, Exercise, Gift)
- transaction_code: VARCHAR(5) (P, S, M, A, D, G, V — raw SEC code)
- transaction_date: DATE
- filing_date: TIMESTAMP
- shares: BIGINT
- price_per_share: DECIMAL(12,4)
- total_value: DECIMAL(16,2)
- shares_owned_after: BIGINT (post-transaction holdings)
- ownership_type: VARCHAR(10) (Direct or Indirect)
- conviction_score: INTEGER (0-100, our calculated score)
- score_breakdown: JSONB (stores individual scoring components)
- price_context: JSONB (stores price change since transaction, 52w high/low, etc.)
- cluster_id: INTEGER (groups related transactions together)
- cluster_type: VARCHAR(30) (classification: "coordinated_buy", "coordinated_sell", "lockup_expiry", "mixed", or null if not in a cluster)
- cluster_metadata: JSONB (stores cluster analysis details — see STEP 4 FACTOR 5 for structure)
- context_tags: TEXT[] (array of tags like "open-market buy", "large trade", "C-suite", etc.)
- created_at: TIMESTAMP DEFAULT NOW()
- expires_at: TIMESTAMP (set to 30 days from created_at for auto-cleanup)

Add an index on (ticker, transaction_date) and (conviction_score DESC) and (cluster_id).

Run a cleanup query on startup that deletes rows where expires_at < NOW().

=== STEP 3: Fetch Form 4 filings from SEC EDGAR using edgartools ===

Create a function called fetch_recent_form4_filings() that:

1. Uses edgartools to get the most recent Form 4 filings:
   from edgar import get_filings
   filings = get_filings(form="4")  # Gets most recent Form 4 filings
   
2. For each filing, parse it into structured data:
   form4 = filing.obj()
   
3. Extract these fields from each parsed filing:
   - filing.company (company name)
   - filing.cik (company CIK number)  
   - form4.reporting_owner_name (insider name)
   - form4.reporting_owner_relationship (dict with isDirector, isOfficer, officerTitle, isTenPercentOwner)
   - form4.transactions (list of transactions with: transaction_date, transaction_code, shares, price_per_share, acquired_disposed_code, shares_owned_following)
   - filing.accession_number (unique filing ID)
   - filing.filing_date

4. For the ticker symbol: The filing object may have it in the issuer data. If not available directly, use the company CIK to look it up from SEC's company tickers JSON:
   https://www.sec.gov/files/company_tickers.json
   Cache this mapping in memory on startup.

5. Filter to only keep transaction codes P (Purchase) and S (Sale) as primary. Also track M (Exercise/Conversion), A (Grant/Award), D (Disposition), G (Gift) but flag them differently.

6. Run this fetch every 2 hours via a background task (use asyncio or APScheduler). Also allow manual trigger via API endpoint.

7. IMPORTANT: Before inserting, check if accession_number already exists in the database. Skip duplicates.

8. Rate limit SEC requests to max 10 requests per second (SEC's guideline). Add a 0.15 second delay between each filing parse.

=== STEP 4: Build the Conviction Scoring Engine ===

Create a function called calculate_conviction_score() that takes a parsed transaction and returns a score 0-100 plus a breakdown dict.

Score these 8 factors (each weighted):

FACTOR 1 — Transaction Size (weight: 15 points max)
- Calculate total_value = shares * price_per_share
- $0-$50K = 2 pts, $50K-$250K = 5 pts, $250K-$1M = 8 pts, $1M-$5M = 12 pts, $5M+ = 15 pts

FACTOR 2 — Insider Title/Role (weight: 20 points max)
- CEO / Chairman = 20 pts
- President / COO = 18 pts
- CFO / CTO / CMO (C-suite) = 16 pts
- SVP / EVP = 12 pts
- VP = 10 pts
- Director = 8 pts
- 10% Owner = 15 pts (these are significant — activist investors, PE firms)
- Other Officer = 6 pts
- Parse the officerTitle string to determine role. Check isDirector, isOfficer, isTenPercentOwner booleans.

FACTOR 3 — Transaction Type (weight: 10 points max)
- Open market purchase (code P, acquired) = 10 pts (strongest signal — they chose to buy with their own money)
- Open market sale (code S) = 7 pts (common but still notable)
- Exercise + Hold (code M, but they keep shares) = 5 pts
- Exercise + Sell (code M immediately followed by S) = 3 pts (weakest — often just cashing out options)
- Gift (code G) = 1 pt
- Grant/Award (code A) = 0 pts (not a voluntary decision)

FACTOR 4 — Price Context (weight: 15 points max)
Use Tradier batch quotes (PRIMARY), Finnhub (FALLBACK 1), or yfinance (FALLBACK 2) to get the stock's current price and 52-week high/low. See STEP 6 below for the full fallback chain and API details.
- Buying after 30%+ drop from 52w high = 15 pts
- Buying after 20-30% drop = 12 pts
- Buying after 10-20% drop = 8 pts
- Buying near 52w low (within 10%) = 10 pts
- Buying near all-time/52w high (within 5%) = 3 pts (less contrarian signal)
- Selling near 52w high = 5 pts (expected)
- Selling after big drop = 12 pts (unusual — bearish signal)
Calculate: pct_from_52w_high = (current_price - high_52w) / high_52w * 100

FACTOR 5 — Clustering (weight: 15 points max) *** THIS IS THE MOST IMPORTANT SIGNAL ***
Clustering is the strongest predictive signal in insider activity. One big trade means nothing — four different insiders loading into the same stock within days is a pattern. Build this factor with extra care.

STEP 5A — Detect clusters:
Query the database to find all transactions for the same ticker within a 14-day rolling window:
SELECT ticker, insider_name, transaction_date, transaction_type, total_value, shares, shares_owned_after, insider_title
FROM insider_transactions 
WHERE ticker = {ticker} 
AND transaction_date BETWEEN {date - 14 days} AND {date + 14 days}
AND accession_number != {current_accession}  -- exclude self

Count DISTINCT insider_name values. If count >= 2, this is a cluster.
Assign the same cluster_id to all transactions in the cluster. Use a sequential integer — increment a counter each time a new cluster is detected.

STEP 5B — Classify cluster type:
For each detected cluster, determine what kind of cluster it is. Store the classification in cluster_type column.

TYPE 1: "coordinated_buy" — Multiple insiders BUYING within 14 days
- All or majority (>70%) of cluster transactions are buys (code P)
- This is the strongest bullish signal. Score HIGH.

TYPE 2: "coordinated_sell" — Multiple insiders SELLING, but NOT a lockup
- All or majority (>70%) of cluster transactions are sells (code S)
- Company has been public for MORE than 180 days (not a recent IPO)
- Sell dates are STAGGERED across multiple days (date spread > 2 days)
- This is a strong bearish signal — "rats fleeing." Score HIGH.

TYPE 3: "lockup_expiry" — Multiple insiders selling, but it's a lockup event
- All or majority (>70%) of cluster transactions are sells (code S)  
- Sell dates are CONCENTRATED on the same day or within 1-2 days of each other
- AND at least one of these conditions:
  a) Company IPO'd within the last 180 days (check if earliest Form 4 filing in our database for this ticker is recent, OR use Finnhub IPO calendar if available)
  b) Transaction sizes are proportionally uniform (each insider selling roughly the same % of their holdings — calculate the standard deviation of position_impact_pct across the cluster. If stdev < 15%, it's likely lockup)
  c) Many of the sellers have insider_title containing "10% Owner" or fund-related titles
- This is a WEAK signal — expected behavior, not unusual. Score LOW.

TYPE 4: "mixed" — Mix of buys and sells from different insiders
- Neither buys nor sells make up >70% of the cluster
- Interesting but ambiguous. Score MEDIUM.

STEP 5C — Score the cluster:
Base score by cluster size:
- 1 insider (no cluster) = 0 pts
- 2 insiders = 5 pts
- 3-4 insiders = 9 pts
- 5-7 insiders = 12 pts
- 8+ insiders = 15 pts

Then apply a cluster_type multiplier:
- "coordinated_buy" = multiply by 1.0 (full score — this is the best signal)
- "coordinated_sell" (not lockup) = multiply by 0.9 (almost as strong as coordinated buy)
- "mixed" = multiply by 0.5 (ambiguous, half credit)
- "lockup_expiry" = multiply by 0.2 (expected behavior, minimal signal)

Round the result. Cap at 15.

STEP 5D — Store cluster metadata in cluster_metadata JSONB column:
{
  "cluster_size": 4,
  "cluster_type": "coordinated_buy",
  "cluster_date_range": {"start": "2026-03-06", "end": "2026-03-16"},
  "date_spread_days": 10,
  "buy_count": 4,
  "sell_count": 0,
  "insiders_in_cluster": ["Cohen Ryan", "Attal Alain", "Cheng Lawrence", "Grube Matt"],
  "roles_in_cluster": ["CEO", "Director", "Director", "VP"],
  "total_cluster_value": 11500000,
  "avg_position_impact_pct": 2.8,
  "position_impact_stdev": 1.2,
  "is_lockup_likely": false,
  "type_multiplier": 1.0,
  "raw_score": 9,
  "final_score": 9
}

STEP 5E — Cluster selling detection heuristics (extra detail for sell clusters):
When a sell cluster is detected, run these additional checks to distinguish "rats fleeing" from routine:

CHECK 1 — Date concentration:
Calculate the spread in days between the earliest and latest transaction in the cluster.
- Same day (spread = 0): likely lockup or coordinated event → lower score
- 2-5 day spread: ambiguous, could go either way
- 6-14 day spread: staggered sells = more likely "rats fleeing" → higher score

CHECK 2 — Role diversity:
Are the sellers from different levels of the org chart?
- All same role (e.g., all Directors) = less unusual, might be board-level planned selling
- Mixed roles (CEO + CFO + Director + VP) = more unusual, suggests broader concern → higher score
Count distinct role categories in the cluster. 3+ distinct roles = add 2 bonus pts.

CHECK 3 — Position impact variance:
Calculate what % of holdings each insider sold.
- Low variance (everyone selling ~same %) = looks coordinated/planned → likely lockup
- High variance (CEO dumps 40%, Director sells 5%) = looks like independent decisions → more unusual
Calculate standard deviation of (shares_sold / shares_owned_after) across cluster members.
- stdev < 10% → likely planned/lockup
- stdev 10-30% → ambiguous
- stdev > 30% → independent decisions, "rats fleeing" pattern

CHECK 4 — Price trend at time of selling:
Using the stock price data (from Tradier/Finnhub/yfinance), check the stock's recent performance:
- Stock is FLAT or slightly declining (within -5% to +2% over prior 30 days) AND insiders are selling = most suspicious. They may know something the market doesn't yet. Add 2 bonus pts.
- Stock is at/near 52w HIGH and insiders selling = expected profit-taking. No bonus.
- Stock already CRASHED (>20% decline) and insiders are STILL selling = very bearish but different signal. Add 1 bonus pt.

Store all check results in the cluster_metadata JSONB so the frontend can display them.

FACTOR 6 — Position Size Impact (weight: 10 points max)
- Calculate: trade_pct = shares_traded / shares_owned_after * 100
- For BUYS: >50% increase in position = 10 pts, 20-50% = 7 pts, 10-20% = 4 pts, <10% = 2 pts
- For SELLS: >50% of holdings sold = 10 pts (major liquidation), 20-50% = 7 pts, 10-20% = 4 pts, <10% = 2 pts

FACTOR 7 — Insider Track Record (weight: 10 points max)
- Look up the same insider_name in our database history (last 30 days of data we have)
- If this insider has filed before: check if their previous buys preceded price increases
- For now (MVP): Simply count how many filings this insider has in our database
- 1 filing (first time we see them) = 5 pts (neutral)
- 2-3 filings = 7 pts (active insider)
- 4+ filings in 30 days = 10 pts (very active — either bullish cluster or regular seller)
- NOTE: Track record accuracy requires months of data. For MVP, just use frequency. Add accuracy tracking later.

FACTOR 8 — Event Proximity (weight: 5 points max)
- Check if transaction date is within 14 days of earnings (use Finnhub earnings calendar endpoint)
- Pre-earnings buy = 5 pts (insider knows something)
- Pre-earnings sell = 4 pts
- No nearby event = 1 pt
- Finnhub endpoint: GET /stock/earnings?symbol={ticker}&from={date-14d}&to={date+14d}

TOTAL SCORE = sum of all 8 factors (max 100)

NOTE ON REBALANCED WEIGHTS: Clustering is now 15 points (up from 10). To keep the total at 100, Transaction Size is reduced from 20 to 15. This reflects the insight that coordination patterns (multiple insiders acting) are more predictive than raw trade size (one insider going big). The full weight distribution is now:
- Transaction Size: 15 pts (was 20)
- Insider Role: 20 pts
- Transaction Type: 10 pts
- Price Context: 15 pts
- Clustering: 15 pts (was 10) *** BOOSTED — strongest signal ***
- Position Impact: 10 pts
- Track Record: 10 pts
- Event Proximity: 5 pts
Total: 100 pts

Store the breakdown as JSONB:
{
  "size": {"score": 10, "max": 15, "detail": "$2.3M total value"},
  "role": {"score": 20, "max": 20, "detail": "CEO"},
  "type": {"score": 10, "max": 10, "detail": "Open market purchase"},
  "context": {"score": 12, "max": 15, "detail": "Buying 25% below 52w high"},
  "cluster": {"score": 9, "max": 15, "detail": "4 insiders buying in 14-day window (coordinated_buy)", "cluster_type": "coordinated_buy", "cluster_size": 4},
  "position_impact": {"score": 7, "max": 10, "detail": "35% increase in holdings"},
  "track_record": {"score": 5, "max": 10, "detail": "First filing in our database"},
  "event_proximity": {"score": 4, "max": 5, "detail": "12 days before earnings"}
}

=== STEP 5: Generate Context Tags ===

For each transaction, generate an array of descriptive tags:

Based on transaction_code:
- P → "open-market buy"
- S → "open-market sell"  
- M → "option exercise"
- A → "grant/award"
- G → "gift"

Based on insider role:
- CEO/Chairman/President → "C-suite"
- Director → "board member"
- 10% Owner → "major shareholder"

Based on value:
- > $1M → "large trade"
- > $10M → "mega trade"

Based on clustering:
- 2+ insiders → "{N} insiders"
- cluster_type = "coordinated_buy" → "coordinated buying"
- cluster_type = "coordinated_sell" AND NOT lockup → "coordinated selling"
- cluster_type = "lockup_expiry" → "lockup expiry"
- cluster_type = "mixed" → "mixed insider activity"
- sell cluster with date_spread > 5 days → "staggered sells"
- sell cluster with 3+ distinct roles → "cross-level selling"

Based on position impact:
- > 50% position change → "major stake change"

Store as TEXT[] array in context_tags column.

=== STEP 6: Price Context Enrichment ===

After scoring, enrich each transaction with price context data. Store in price_context JSONB column:
{
  "price_at_filing": 28.26,
  "current_price": 27.42,
  "change_since_filing_pct": -2.97,
  "high_52w": 44.50,
  "low_52w": 22.10,
  "pct_from_52w_high": -38.4,
  "return_30d": null,  // fill in later via scheduled job
  "return_90d": null,
  "vs_52w_high": "-38.4%"
}

Use a 3-tier fallback chain for stock price data:

PRIMARY — Tradier API (120 requests/min, supports batch quotes, returns 52-week high/low natively):
The Tradier quotes endpoint accepts MULTIPLE symbols in one call (comma-separated), which is critical for efficiency.
Use the sandbox endpoint for delayed data (free, no brokerage account needed):
GET https://sandbox.tradier.com/v1/markets/quotes?symbols={ticker1},{ticker2},{ticker3}
Headers: Authorization: Bearer {TRADIER_API_KEY}, Accept: application/json

The Tradier quote response includes these fields we need:
- last: current price
- week_52_high: 52-week high
- week_52_low: 52-week low  
- change_percentage: daily change %
- prevclose: previous close

BATCH STRATEGY: Collect all unique tickers from the current batch of insider filings, then make ONE Tradier call with up to 50 symbols comma-separated. This means enriching 50 transactions costs only 1 API call instead of 50.

Example: GET https://sandbox.tradier.com/v1/markets/quotes?symbols=AAPL,TSLA,GME,DASH,WBD

If you have a Tradier production token (from a brokerage account), use https://api.tradier.com/v1/markets/quotes instead for real-time data. The sandbox gives 15-minute delayed data which is fine for this use case.

FALLBACK 1 — Finnhub (60 requests/min, one symbol at a time):
GET https://finnhub.io/api/v1/quote?symbol={ticker}&token={FINNHUB_API_KEY}
Returns: c (current price), h (high), l (low), o (open), pc (previous close)
NOTE: Finnhub quote does NOT include 52-week high/low. If Finnhub is needed, also call:
GET https://finnhub.io/api/v1/stock/metric?symbol={ticker}&metric=all&token={FINNHUB_API_KEY}
This returns 52WeekHigh and 52WeekLow in the "metric" object.

FALLBACK 2 — yfinance (completely free, no API key, no rate limit, but slower):
Install: pip install yfinance --break-system-packages
import yfinance as yf
ticker_data = yf.Ticker("AAPL")
info = ticker_data.info
current_price = info.get("currentPrice") or info.get("regularMarketPrice")
high_52w = info.get("fiftyTwoWeekHigh")
low_52w = info.get("fiftyTwoWeekLow")

yfinance is slower (~1-2 seconds per ticker) but has NO rate limits and NO API key. Use it as the last resort when Tradier and Finnhub both fail. It also returns sector, industry, and company name which can be useful for enrichment.

IMPORTANT: Always try Tradier first (batch call = fast), then Finnhub for any tickers that failed, then yfinance for any remaining failures. Handle all API errors gracefully — if ALL three fail for a ticker, just save the transaction with null price context. The scoring engine should still calculate the other 7 factors without price data.

=== STEP 9: Background Scheduler ===

Set up a background task that runs every 2 hours to:
1. Fetch new Form 4 filings from SEC EDGAR
2. Score each new transaction
3. Enrich with price context
4. Insert into database (skip duplicates via accession_number UNIQUE constraint)
5. Update cluster_ids for any new clusters detected
6. Clean up expired rows (expires_at < NOW())

Use APScheduler or asyncio with the FastAPI lifespan:
from apscheduler.schedulers.asyncio import AsyncIOScheduler
