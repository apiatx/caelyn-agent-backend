Implement frontend UI support for upgraded history and cross-model conversation continuity in CaelynAI.

You only have access to the FRONTEND repo.

Context:
- The backend already owns persistence, conversation continuity, ticker snapshot capture, and APIs.
- The frontend should treat backend as source of truth.
- Do not invent parallel conversation/thread logic on the client.

Backend behavior already implemented:
1. Conversation continuity:
   - conversation_id is canonical
   - when conv_id is valid, backend DB history is source of truth
   - switching models does NOT create a new conversation automatically
   - a conversation started with one model can be continued by another model using the same conv_id

2. Backend endpoints now available:
   - GET /api/history/sidebar
     returns only the 5 most recent conversations for the right-side quick-reference panel
   - GET /api/history/page
     returns conversation-level history data for the full History page, including ticker mention snapshots
   - POST /api/prices/batch
     accepts tickers and returns current prices
   - GET /api/conversations/{conv_id}
     returns a conversation thread and messages; messages may include ticker_mentions

3. Ticker mention data:
   - mention-time price is persisted by backend in ticker_mentions
   - old conversations may have ticker_mentions: []
   - frontend should treat persisted mention_price as historical source of truth
   - frontend may use /api/prices/batch for current prices when viewing history

## Main frontend goals

### 1) Full History page
Update the full History page so it shows, where available:
- conversation title / metadata
- message context
- model used
- ticker symbol
- mention-time price from backend
- current price from /api/prices/batch
- % gain/loss since mention-time price

Requirements:
- mention-time price must come from backend persisted data
- current price should be fetched on page load / view load
- compute % gain/loss in frontend using mention_price vs current price
- deduplicate ticker requests when possible
- fail gracefully if some current prices are unavailable
- old history with no ticker_mentions should still render cleanly

### 2) Right-side terminal history panel
Implement/fix the designated right-side history area so it:
- renders in the intended right-side terminal location
- uses GET /api/history/sidebar
- shows only the 5 most recent conversations
- acts as quick reference only
- does NOT show ticker backtesting datapoints
- allows click-to-load/open a conversation

Display only lightweight fields such as:
- title or preview
- last updated timestamp
- optional last_model_used if useful

### 3) Preserve conversation thread on model switch
Update frontend state management so that:
- active conversation_id is separate from selected model
- switching models does NOT wipe the active thread
- switching to Caelyn mode does NOT create a new conversation unless user explicitly starts New Chat
- the next message continues the same conversation_id
- loading an existing conversation restores that thread regardless of which model originally created it

Important:
- do not send stale client-side history that overrides backend thread state
- when conv_id exists, backend history is source of truth
- frontend should not try to rebuild conversation continuity on its own
